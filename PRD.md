## 三、文档说明

### 3.1 名词解释

| 术语 | 说明 |
|------|------|
| 基建 | 广告投放中的"创建广告、复制广告、调整预算"等基础运营动作 |
| 生命周期 | 商品或广告从上线到衰退的全过程阶段 |
| ROI | Return On Investment，投资回报率 = 收入 / 成本 |
| 冷启动 | 新广告上线后能否持续消耗并产生回报的阶段 |
| 决策 | 系统输出的"应该做什么"的判断，不是实际执行动作 |
| 饱和攻击 | 短时间内大量创建广告，争取曝光的策略 |

| ROI | Return On Investment | 投资回报率 = (d0_order_amt + d0_ad_amt) / cost |

| d0_ad_amt | D0广告收入 | 广告分成收入 |

## 四、需求背景

### 4.1 业务问题

当前广告投放中，"基建工作"面临三大痛点：

| 痛点 | 表现 | 影响 |
|------|------|------|
| 被动响应 | 优化师在广告效果衰减后才开始补充新广告 | 预算浪费、效果下滑 |
| 经验依赖 | 何时新建、复制多少、启用哪些素材，依赖个人经验 | 决策质量参差不齐 |
| 效率瓶颈 | 批量创建工具存在，但"创建什么"仍需人工决策 | 人效低 |

### 4.2 解决方案

智能基建系统将"人工定期批量创建"升级为"系统实时自动耕作"。

**核心定位**：系统作为**广告投放指挥官**，只输出决策，不实际执行。

**关键发现**（基于数据分析）：
- 收入 = d0_order_amt + d0_ad_amt
- **盈利标准：ROI > 40%**
- 盈利Campaign的订单收入占比 > 90%，广告只是放大器
- **72小时是判断Campaign生死的关键节点**

### 4.3 预期收益

| 指标维度 | 预期效果 | 对应目标 |
|----------|----------|----------|
| 基建效率 | 优化师人均管理广告数提升 200%+ | 提高人效 |
| 基建成功率 | 新广告渡过冷启动期成功率提升 30% | 优化投放表现 |
| 账户健康度 | 衰退期广告占比降低，成长期广告占比提升 | 优化投放表现 |
| 素材利用率 | 优质素材利用周期延长，库存周转率提升 50% | 降本增效 |

### 5.2 功能详细说明

#### 5.2.1 生命周期判定

**功能描述**：根据广告实体的ROI数据，自动判断其所处的生命周期阶段。

**核心指标**：
```
收入 = d0_order_amt + d0_ad_amt
ROI = 收入 / 成本
盈利标准 = ROI > 40%
```

**关键数据发现**：
- 72小时是判断Campaign生死的关键节点
- 前24h ROI < 10% 的Campaign，85%最终无法盈利
- 盈利Campaign的订单收入占比 > 90%
- 49%的存活>7天的Campaign能盈利

**Campaign生命周期阶段（基于ROI）**：

| 阶段 | 判定条件 | 数据依据 | 置信度 | 运营建议 |
|------|----------|----------|--------|----------|
| 待观察 | 投放 < 24h | 时间不足，无法判断 | 0.50 | 等待 |
| 冷死亡 | 投放 > 72h 且从未产生收入 | 24.3%的Campaign从未产生收入 | 0.95 | 直接放弃 |
| 冷启动 | 前24h ROI < 10% | 前24h低ROI的85%最终不盈利 | 0.85 | 观察，24h决策 |
| 验证期 | 24-72h ROI在10-40% | 关键决策点，过了的30%能盈利 | 0.75 | 持续观察 |
| 成长期 | 72h后 ROI > 40% | 92%的这类Campaign保持盈利 | 0.85 | 加预算 |
| 持续盈利 | ROI > 40% 超过7天 | 97%最终盈利 | 0.90 | 重点维护 |
| 衰退期 | ROI从高点下降 > 50% | 后期ROI显著下降 | 0.85 | 准备替代 |
| 关停期 | ROI < 10% 持续72h+ | 几乎无法翻盘 | 0.90 | 关停 |

**商品（ShortPlay）生命周期阶段（基于回测数据验证）**：

判定逻辑基于时间段 + ROI指标组合，时间段划分如下：

| 时间段 | 阶段 | 判定条件 | 精确率 |
|--------|------|----------|--------|
| < 3天 | 待观察 | 时间不足，无法判断 | - |
| 3-7天 | 入场期 | 近3天ROI均值 > 40% | 87% |
| 3-7天 | 退出期 | 近3天ROI均值 < 10% | 85% |
| 7-14天 | 衰退期 | ROI下滑 > 30% | 80% |
| 7-14天 | 成长期 | ROI > 40% 且近3天均值 > 前3天均值 × 1.2（上升20%） | 87% |
| 7-14天 | 稳定期 | ROI > 40% 但趋势平稳 | 86% |
| > 14天 | 退出期 | ROI < 10% 持续5天 | 85% |
| > 14天 | 衰退期 | ROI下滑 > 30% | 80% |
| > 14天 | 成长期 | ROI > 40% 且趋势上升 | 87% |
| > 14天 | 稳定期 | 近5天ROI都在30%-80% | 86% |
| - | 无投放 | 成本 = 0 | - |

**趋势指标定义**：
- 近3天ROI均值: `sum(history[-3:]) / 3`
- 前3天均值: `sum(history[-6:-3]) / 3`（用于趋势对比）
- 趋势上升: `recent_3_avg > previous_3_avg × 1.2`
- ROI下滑: `(first_half_avg - second_half_avg) / first_half_avg > 30%`

#### 5.2.2 基建策略管理

**功能描述**：配置"什么阶段触发什么动作"的规则，支持前端页面可视化配置。

**策略规则结构**：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| 规则名称 | string | 是 | 自定义名称，如"冷启动补量策略" |
| 描述 | string | 否 | 规则用途说明 |
| 维度 | enum | 是 | product / campaign |
| 触发阶段 | list | 是 | 可多选 |
| 执行动作 | enum | 是 | 具体动作类型 |
| 执行规模 | int | 是 | 每次触发生成的广告数量上限 |
| 最小置信度 | float | 否 | 默认0.7，只有检测置信度高于此值才触发 |
| 冷却时间 | int | 否 | 默认24小时，同一实体需等待N小时才能再次触发 |
| 优先级 | int | 否 | 默认100，数字越小优先级越高 |
| 额外条件 | list | 否 | 除了阶段外还需满足的指标条件 |
| 状态 | bool | 否 | 默认启用 |

**支持的执行动作（基于ROI分析）**：

| 动作 | 代码 | 说明 | 适用阶段 |
|------|------|------|----------|
| 饱和式攻击 | GROWTH_BURST | 短时间内大量创建广告 | 冷死亡 |
| 增加预算 | INCREASE_BUDGET | 提高预算扩大规模 | 成长期、持续盈利 |
| 渠道扩张 | CHANNEL_EXPAND | 将成功的商品+素材复制到其他渠道 | 成长期、持续盈利 |
| 复制广告 | CLONE_AD | 复制现有广告进行测试 | 冷启动 |
| 素材预热 | MATERIAL_PREPARE | 用小预算测试新素材潜力 | 验证期 |
| 预算平滑 | BUDGET_SMOOTH | 调整不同广告间的预算分配 | 验证期 |
| 维持现状 | MAINTAIN | 保持当前投放策略 | 持续盈利 |
| 有序关停 | GRACEFUL_SHUTDOWN | 逐步降低预算后关停 | 衰退期、关停期 |
| 基建补充 | REBUILD | 用新素材重新创建广告 | 衰退期、亏损 |
| 降低预算 | REDUCE_BUDGET | 减少预算开支 | 衰退期 |

| 冷启动-复制替换 | Campaign | 冷启动 | 复制广告 | 10 | 12h |

#### 5.2.3 决策输出

**功能描述**：将策略匹配结果转换为可执行的决策，并存储到日志文件。

**决策格式**：

```json
{
    "timestamp": "2026-04-21T10:30:00",
    "entity_id": "campaign_001",
    "dimension": "campaign",
    "stage": "campaign_cold_dead",
    "action": "GROWTH_BURST",
    "scale": 50,
    "confidence": 0.855,
    "reason": "阶段campaign_cold_dead触发规则冷死亡-饱和攻击"
}
```

**存储位置**：

```
logs/decisions/YYYY-MM-DD/
├── decisions.jsonl           # 每日决策汇总（追加写入）
└── decision_HHMMSS.json      # 单条决策详情
```

**决策统计指标**：

| 指标 | 说明 |
|------|------|
| 今日决策数 | 当日生成的决策总条数 |
| 自动化渗透率 | 系统自动创建的广告占总新建广告的比例 |
| 动作分布 | 各种动作类型的执行次数 |

### 5.3 系统运行流程

#### 5.3.1 整体流程

```
┌─────────────────────────────────────────────────────────────────────┐
│                         完整业务流程                                  │
└─────────────────────────────────────────────────────────────────────┘

Step 1: 数据输入
─────────────────────────────────────────
Hive数仓 ──定时拉取──▶ 解析聚合 ──▶ ROI指标计算
                                       │
                                       ▼
Step 2: 生命周期判定（基于ROI）
─────────────────────────────────────────
                                   ROI指标
                                      │
                              ┌───────┴───────┐
                              ▼               ▼
                        商品维度         Campaign维度
                              │               │
                        按ROI阈值判断      按ROI阈值判断
                              │               │
                              └───────┬───────┘
                                      ▼
                              阶段 + 置信度 + 盈利概率

Step 3: 策略匹配
─────────────────────────────────────────
                              当前阶段
                                      │
                              ┌───────┴───────┐
                              ▼               ▼
                         查询规则         条件过滤
                              │               │
                              │         冷却时间检查
                              │               │
                              │         ROI条件检查
                              └───────┬───────┘
                                      ▼
                               匹配的策略

Step 4: 决策输出
─────────────────────────────────────────
                              匹配的策略
                                      │
                              生成决策JSON
                                      │
                              写入日志文件
                                      │
                              看板展示统计
```

#### 5.3.2 策略匹配逻辑

```
输入：entity_id, dimension, current_stage, confidence, metrics

1. 查询所有启用的规则

2. 维度过滤
   └── dimension == rule.dimension

3. 阶段过滤
   └── current_stage in rule.trigger_stages

4. 置信度过滤
   └── confidence >= rule.confidence_min

5. 冷却时间检查
   └── 当前时间 - 上次触发时间 >= cooldown_hours

6. ROI条件检查（可选）
   └── metrics.roi >= rule.roi_min

7. 按优先级排序输出
```

#### 5.3.3 典型业务场景

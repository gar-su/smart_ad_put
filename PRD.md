## 三、文档说明

### 3.1 名词解释

| 术语 | 说明 |
|------|------|
| 基建 | 广告投放中的"创建广告、复制广告、调整预算"等基础运营动作 |
| 生命周期 | 商品或广告从上线到衰退的全过程阶段 |
| 冷启动 | 新广告上线后能否持续消耗的阶段 |
| 决策 | 系统输出的"应该做什么"的判断，不是实际执行动作 |
| 饱和攻击 | 短时间内大量创建广告，争取曝光的策略 |

### 3.2 术语 / 缩略词

| 缩略词 | 全称 | 说明 |
|--------|------|------|
| Product | 商品 | 指短剧/内容商品 |
| Campaign | 广告单元 | 广告投放计划 |
| CTR | Click Through Rate | 点击率 |
| CPA | Cost Per Action | 每次转化成本 |
| ROI | Return On Investment | 投资回报率 |
| TDD | Test Driven Development | 测试驱动开发 |

## 四、需求背景

### 4.1 业务问题

当前广告投放中，"基建工作"面临三大痛点：

| 痛点 | 表现 | 影响 |
|------|------|------|
| 被动响应 | 优化师在广告效果衰减后才开始补充新广告 | 预算浪费、效果下滑 |
| 经验依赖 | 何时新建、复制多少、启用哪些素材，依赖个人经验 | 决策质量参差不齐 |
| 效率瓶颈 | 批量创建工具存在，但"创建什么"仍需人工决策 | 人效低 |

### 4.2 解决方案

智能基建系统将"人工定期批量创建"升级为"系统实时自动耕作"。

**核心定位**：系统作为**广告投放指挥官**，只输出决策，不实际执行。

```
人工决策 → 系统决策
"我觉得应该新建广告" → "数据显示应触发饱和攻击策略，建议创建50条"
```

### 4.3 预期收益

| 指标维度 | 预期效果 | 对应目标 |
|----------|----------|----------|
| 基建效率 | 优化师人均管理广告数提升 200%+ | 提高人效 |
| 基建成功率 | 新广告渡过冷启动期成功率提升 30% | 优化投放表现 |
| 账户健康度 | 衰退期广告占比降低，成长期广告占比提升 | 优化投放表现 |
| 素材利用率 | 优质素材利用周期延长，库存周转率提升 50% | 降本增效 |

### 5.2 功能详细说明

#### 5.2.1 生命周期判定

**功能描述**：根据广告实体的指标数据，自动判断其所处的生命周期阶段。

**支持的维度**：

| 维度 | 说明 | 适用实体 |
|------|------|----------|
| 商品（Product） | 短剧级别的生命周期 | short_play_id |
| 广告单元（Campaign） | Campaign级别的生命周期 | campaign_id |

**商品生命周期阶段**：

| 阶段 | 判定条件 | 置信度 | 运营建议 |
|------|----------|--------|----------|
| 冷启动失败 | duration ≤ 24h 且 cost < 50元 | 0.9 | 立即补量 |
| 引入期 | 其他情况 | 0.6 | 持续观察 |
| 成长期 | duration > 24h 且 cost_change > +20% | 0.85 | 保持投放，可扩量 |
| 成熟期 | -20% ≤ cost_change ≤ +20% 且 cost ≥ 200元 | 0.8 | 维持现状 |
| 衰退期 | cost_change < -30% | 0.9 | 准备替代品 |

**Campaign生命周期阶段**：

| 阶段 | 判定条件 | 置信度 | 运营建议 |
|------|----------|--------|----------|
| 冷死亡 | duration ≤ 24h 且 cost < 50元 | 0.95 | 直接放弃 |
| 冷启动 | duration ≤ 24h 且 cost ≥ 50元 | 0.85 | 持续观察 |
| 增长期 | cost_change > +20% | 0.8 | 加预算 |
| 稳定期 | -30% ≤ cost_change ≤ +30% | 0.75 | 维持 |
| 衰退期 | cost_change < -30% | 0.85 | 准备替代 |
| 关停期 | cost/hour < 1元 且 duration > 48h | 0.9 | 关停 |

**输入参数**：

| 参数 | 类型 | 说明 |
|------|------|------|
| entity_id | string | 实体唯一标识 |
| duration_hours | float | 投放时长（小时） |
| cost_first_24h | float | 首日消耗 |
| cost_last_24h | float | 最后24小时消耗 |
| cost_change_pct | float | 消耗变化百分比（如 0.20 = +20%） |
| ctr_first_24h | float | 首日CTR（可选） |
| total_pays | int | 总付费数 |

**输出参数**：

| 参数 | 类型 | 说明 |
|------|------|------|
| entity_id | string | 实体唯一标识 |
| stage | string | 判定阶段 |
| confidence | float | 判定置信度 |
| reason | string | 判定原因说明 |
| survival_probability | float | 基于首日消耗的存活概率 |

**原型/示意图**：

```
┌─────────────────────────────────────────────────────────────┐
│  生命周期判定                                               │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  输入：entity_id=short_play_001                             │
│       duration_hours=20                                     │
│       cost_first_24h=30                                    │
│       cost_change_pct=0                                     │
│                                                             │
│  ────────────────────────────────────────────────────────   │
│                                                             │
│  输出：stage=product_cold_start                             │
│       confidence=0.9                                        │
│       reason="冷启动失败: 投放20h, 首日消耗30元 < 50元"   │
│       survival_probability=0.68                              │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

#### 5.2.2 基建策略管理

**功能描述**：配置"什么阶段触发什么动作"的规则，支持前端页面可视化配置。

**策略规则结构**：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| 规则名称 | string | 是 | 自定义名称，如"冷启动补量策略" |
| 描述 | string | 否 | 规则用途说明 |
| 维度 | enum | 是 | product / campaign |
| 触发阶段 | list | 是 | 可多选，如["冷启动失败", "衰退期"] |
| 执行动作 | enum | 是 | 具体动作类型 |
| 执行规模 | int | 是 | 每次触发生成的广告数量上限 |
| 最小置信度 | float | 否 | 默认0.7，只有检测置信度高于此值才触发 |
| 冷却时间 | int | 否 | 默认24小时，同一实体需等待N小时才能再次触发 |
| 优先级 | int | 否 | 默认100，数字越小优先级越高 |
| 额外条件 | list | 否 | 除了阶段外还需满足的指标条件 |
| 状态 | bool | 否 | 默认启用 |

**支持的执行动作**：

| 动作 | 代码 | 说明 |
|------|------|------|
| 饱和式攻击 | GROWTH_BURST | 短时间内大量创建广告 |
| 渠道扩张 | CHANNEL_EXPAND | 将成功的商品+素材复制到其他渠道 |
| 复制广告 | CLONE_AD | 复制现有广告进行测试 |
| 预算平滑 | BUDGET_SMOOTH | 调整不同广告间的预算分配 |
| 素材预热 | MATERIAL_PREPARE | 用小预算测试新素材潜力 |
| 有序关停 | GRACEFUL_SHUTDOWN | 逐步降低预算后关停 |
| 基建补充 | REBUILD | 用新素材重新创建广告 |

**预置策略模板**：

| 模板名称 | 维度 | 触发阶段 | 执行动作 | 规模 | 冷却 |
|----------|------|----------|----------|------|------|
| 冷启动失败-饱和攻击 | 商品 | 冷启动失败 | 饱和式攻击 | 50 | 24h |
| 衰退期-基建补充 | 商品 | 衰退期 | 基建补充 | 30 | 72h |
| 成长期-渠道扩张 | 商品 | 成长期 | 渠道扩张 | 20 | 48h |
| 冷死亡-复制替换 | Campaign | 冷死亡 | 复制广告 | 5 | 12h |
| 衰退期-有序关停 | Campaign | 衰退期 | 有序关停 | 1 | 24h |
| 关停期-素材预热 | Campaign | 关停期 | 素材预热 | 3 | 48h |

**原型/示意图**：

├─────────────────────────────────────────────────────────────────────┤

#### 5.2.3 决策输出

**功能描述**：将策略匹配结果转换为可执行的决策，并存储到日志文件。

**决策格式**：

```json
{
    "timestamp": "2026-04-20T10:30:00",
    "entity_id": "short_play_001",
    "dimension": "product",
    "stage": "product_cold_start",
    "action": "GROWTH_BURST",
    "scale": 50,
    "reason": "冷启动失败: 首日消耗30元 < 50元"
}
```

**存储位置**：

```
logs/decisions/YYYY-MM-DD/
├── decisions.jsonl           # 每日决策汇总（追加写入）
└── decision_HHMMSS.json      # 单条决策详情
```

**决策统计指标**：

| 指标 | 说明 |
|------|------|
| 今日决策数 | 当日生成的决策总条数 |
| 自动化渗透率 | 系统自动创建的广告占总新建广告的比例 |
| 动作分布 | 各种动作类型的执行次数 |

**原型/示意图**：

### 5.3 系统运行流程

#### 5.3.1 整体流程

```
┌─────────────────────────────────────────────────────────────────────┐
│                         完整业务流程                                  │
└─────────────────────────────────────────────────────────────────────┘

Step 1: 数据输入
─────────────────────────────────────────
Hive数仓 ──定时拉取──▶ 解析聚合 ──▶ 实体指标
                                      │
                                      ▼
Step 2: 生命周期判定
─────────────────────────────────────────
                                   实体指标
                                      │
                              ┌───────┴───────┐
                              ▼               ▼
                        商品维度         Campaign维度
                              │               │
                        按阈值判断        按阈值判断
                              │               │
                              └───────┬───────┘
                                      ▼
                              阶段 + 置信度

Step 3: 策略匹配
─────────────────────────────────────────
                              当前阶段
                                      │
                              ┌───────┴───────┐
                              ▼               ▼
                         查询规则         条件过滤
                              │               │
                              │         冷却时间检查
                              │               │
                              │         额外条件检查
                              └───────┬───────┘
                                      ▼
                               匹配的策略

Step 4: 决策输出
─────────────────────────────────────────
                              匹配的策略
                                      │
                              生成决策JSON
                                      │
                              写入日志文件
                                      │
                              看板展示统计
```

#### 5.3.2 策略匹配逻辑

```
输入：entity_id, dimension, current_stage, confidence, metrics

1. 查询所有启用的规则

2. 维度过滤
   └── dimension == rule.dimension

3. 阶段过滤
   └── current_stage in rule.trigger_stages

4. 置信度过滤
   └── confidence >= rule.confidence_min

5. 冷却时间检查
   └── 当前时间 - 上次触发时间 >= cooldown_hours

6. 额外条件检查（可选）
   └── all(condition in metrics for condition in rule.conditions)

7. 按优先级排序输出
```
